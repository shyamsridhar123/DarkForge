# opensandbox-azure

Python SDK for [OpenSandbox on Azure](https://github.com/your-org/opensandbox).

## Install

```bash
pip install opensandbox-azure
```

## Quickstart

```python
from opensandbox_azure import SandboxClient
from azure.identity import DefaultAzureCredential

client = SandboxClient(
    api_url="https://api-opensandbox.example.com",
    credential=DefaultAzureCredential(),
    scope="api://<api-app-id>/.default",   # App ID URI of the control-plane app registration
)

sess = client.create_session(image="acr.example.azurecr.io/sandbox/base/python:3.12")
try:
    result = sess.run("python -c 'print(\"hello\")'")
    print(result.stdout)       # hello
    print(result.exit_code)    # 0
finally:
    sess.delete()
```

## Authentication

The SDK delegates token acquisition entirely to [`azure-identity`](https://pypi.org/project/azure-identity/) — no hand-rolled MSAL.

### DefaultAzureCredential (recommended for production)

```python
from azure.identity import DefaultAzureCredential

client = SandboxClient(
    api_url="...",
    credential=DefaultAzureCredential(),
    scope="api://<api-app-id>/.default",
)
```

`DefaultAzureCredential` tries, in order: environment variables → Workload Identity → Managed Identity → Azure CLI → Visual Studio Code → Interactive browser. See [azure-identity docs](https://learn.microsoft.com/python/api/azure-identity/azure.identity.defaultazurecredential).

### InteractiveBrowserCredential (local development)

```python
from azure.identity import InteractiveBrowserCredential

client = SandboxClient(
    api_url="...",
    credential=InteractiveBrowserCredential(),
    scope="api://<api-app-id>/.default",
)
```

A browser tab opens on first token acquisition.

### Scope format

Always pass the full scope string:

```
api://<api-app-id>/.default
```

where `<api-app-id>` is the **client ID** (GUID) of the OpenSandbox control-plane App Registration. This sets `aud` in the issued JWT to the correct value for the OBO exchange.

## Async API

Every method has an `a`-prefixed async counterpart:

```python
import asyncio
from opensandbox_azure import SandboxClient
from azure.identity import DefaultAzureCredential

async def main():
    client = SandboxClient(
        api_url="https://api-opensandbox.example.com",
        credential=DefaultAzureCredential(),
        scope="api://<api-app-id>/.default",
    )
    sess = await client.acreate_session(image="acr.example.azurecr.io/sandbox/base/python:3.12")
    try:
        result = await sess.arun("echo hello")
        print(result.stdout)
    finally:
        await sess.adelete()

asyncio.run(main())
```

## Low-latency tier

The default tier creates a fresh pod per session with a **per-user UAMI** (full identity isolation). For latency-sensitive workloads you can opt into a shared warm-pool pod:

```python
sess = client.create_session(
    image="acr.example.azurecr.io/sandbox/base/python:3.12",
    low_latency=True,
)
```

> **Role requirement:** the calling user must have the `SandboxLowLatency` Entra role assignment on the OpenSandbox application. Without it the server returns `403 AuthorizationError`. Shared-pool sessions use a control-plane UAMI (not the user's own UAMI) and are rate-limited; all commands are audit-logged with the user's OID as a claim.

## Error handling

```python
from opensandbox_azure.exceptions import (
    AuthenticationError,   # 401 — missing/invalid token or OBO failure
    AuthorizationError,    # 403 — valid identity, insufficient permissions
    RateLimitError,        # 429 — too many requests
    SessionNotFoundError,  # 404 — session ID not found
    PropagationTimeoutError,  # 503 — workload-identity FC propagation race; retry
    SandboxError,          # base class
)

try:
    sess = client.create_session(image="...")
except AuthenticationError:
    print("Check your credential / scope.")
except PropagationTimeoutError as exc:
    print(f"Retry after {exc.retry_after}s")
except SandboxError as exc:
    print(f"HTTP {exc.status_code}: {exc}")
```

## Distributed tracing

Every outbound request carries a [W3C `traceparent`](https://www.w3.org/TR/trace-context/) header generated by the SDK. This allows correlating a single user action across:

- ACA control-plane (Application Insights)
- AKS API server (`AKSAudit` log)
- Cilium Hubble flow logs
- Execution-daemon stdout

To propagate an existing trace context, set the `TRACEPARENT` environment variable before making calls.
